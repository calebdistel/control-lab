## Hook

You write code to count how many game pieces the robot has collected. You test it. The robot grabs ten pieces. Every log line says `Collected: 1`. It never gets past 1.

```java
void periodic() {
    int count = 0;
    if (sensor.hasGamePiece()) {
        count = count + 1;
        System.out.println("Collected: " + count);
    }
}
```

`periodic()` runs 50 times per second. Each call recreates `count` from scratch. The robot has no memory of what happened 20 milliseconds ago.

## Core concept

<Key>
Where you declare a variable determines how long it lives. Variables inside a method are created fresh every call. Variables declared as class fields persist for the life of the object.
</Key>

## Walk-through

The fix: declare `count` outside `periodic()`, as a field on the class.

```java
int count = 0;          // field — lives on the object

void periodic() {
    if (sensor.hasGamePiece()) {
        count = count + 1;
        System.out.println("Collected: " + count);
    }
}
```

Now `count` is created once and survives every call to `periodic()`.

This is the first design decision you make about every variable: *how long does this value need to live?*

| Situation | Declare as |
|-----------|------------|
| Only needed inside one method call | Local variable |
| Needs to survive between loop calls | Class field |
| Needs to survive across subsystem resets | Static field or external state |

<Note>
In WPILib, your Subsystem class is instantiated once at startup. Fields on a Subsystem live for the entire match.
</Note>

## Try it yourself

void periodic() {

}

## Key takeaways

- Local variables are created fresh every time a method is called.
- Class fields persist for the life of the object — across every `periodic()` call.
- Most robot state (positions, counts, modes) belongs in fields.
- Before declaring a variable, ask: does this value need to outlive this function call?

## Common confusions

**"Why not just make everything a field?"** Local variables communicate intent — "this only matters right now." Overusing fields makes it harder to reason about what the robot is remembering and when.

**"My field keeps resetting when I redeploy."** Redeploying reboots the robot. Fields reset on startup. If you need something to survive a reboot, it has to be written to disk or read from a sensor.
